# H+ Analytics — Slide Content: Research, Generation, Validation

This skill owns three responsibilities for every text field on a property slide deck:

1. **Sourcing** — where the value comes from, in priority order.
2. **Generation** — deterministic templates and the LLM contract that produces investor-grade copy.
3. **Validation** — char-limit enforcement and budget realism for any proposed transformation.

The deterministic templates below are also the LLM's formatting guide and its fallback when the LLM is unavailable.

---

## Source Priority (Research Pipeline)

For every field, resolve in this order. Stop at the first source that returns a value passing validation. This applies to `property-vision.ts` (`generatePropertyVisionText`) and any future slide-content generator.

| # | Source | When it wins | Notes |
|---|---|---|---|
| 1 | **Stored property data** | The H+ DB has an explicit, non-empty value for that field (e.g. `properties.description`, structured amenities, `roomCount`, `startAdr`, `purchasePrice`). | Never overwrite a curated value with a generated one. If the stored value violates the char limit, truncate at a word boundary — do not regenerate. |
| 2 | **`hplus-renovation-benchmarks` skill** | The field is a deterministic derivation: per-key cost ranges, market ADR, comp-set occupancy, transformation cost lines, asset-type labels. | These are sourced numbers (HVS, STR, Turner) — prefer them over LLM guesses. |
| 3 | **LLM with web-research cues** | The field needs market-specific narrative the DB and benchmarks don't carry: city tourism stats, named demand drivers ("Belleayre ski area"), nearby anchor institutions, recent comparable openings, county-level visitor counts. | Use Anthropic Claude (the model pinned in `property-vision.ts`). The prompt MUST list the property's location and ask for citation-ready facts; the LLM may surface concrete public stats from training data. Do not invent figures — round, attribute by region, or fall through to source 4. |
| 4 | **Deterministic templates** (this file, `TEMPLATES` block below) | The LLM is unavailable, returns invalid JSON, fails validation, or proposes content the budget realism check rejects. | Always available, always within char limits, always within budget envelope. |

**Implementation contract.** A slide-content generator is correct iff:

- It records which source produced each field (for debugging — log only, do not surface to investor).
- A validation failure at source N falls through to source N+1, never silently inserts the failed value.
- The merged result has every field populated and every field within its Max Length.

## Char-Limit Enforcement

Before any value is returned to the PPTX generator (Track 1) or the satori renderer (Track 2), run this validator. The merge step at the bottom of `generatePropertyVisionText` is the canonical insertion point.

```ts
const MAX = {
  cinematicCaption: 60,
  visionHeadline: 70,
  visionBullet1: 80, visionBullet2: 80,
  badgeText: 35,
  descriptionParagraph: 180,
  operationalModelText: 80,
  revenueBullet: 80, programmingBullet: 80,
  operationalParagraph: 200,
  investmentModelConcept: 120,
  marketRationale: 120,
  reason1Label: 60, reason1Detail: 100,
  reason2Label: 60, reason2Detail: 100,
  reason3Label: 60, reason3Detail: 100,
  closingLine: 120,
  transformationDescription: 200,
} as const;

function fitToLimit(value: string, max: number): string {
  const v = value.trim().replace(/\s+/g, " ");
  if (v.length <= max) return v;
  // word-boundary truncation, drop trailing punctuation, append ellipsis only if it fits
  const cut = v.slice(0, max).replace(/\s+\S*$/, "").replace(/[,;:.\-–—]+$/, "");
  return cut.length + 1 <= max ? cut + "…" : cut;
}
```

**Rules:**

1. **Always fit, never reject.** A field that exceeds Max gets truncated, never dropped — a missing field would break the slide layout worse than a truncated one.
2. **LLM regeneration on overflow is allowed once.** If the LLM returns a value > Max + 30%, retry the LLM with an explicit "previous value was N chars, max is M chars" instruction before truncating. Do not loop more than once.
3. **No HTML, no Markdown.** Strip backticks, asterisks, and any tag-like sequence before length-checking — PPTX text frames render them literally.
4. **ALL CAPS fields** (`cinematicCaption`, `badgeText`): uppercase AFTER truncation, not before, so character-count math stays simple.
5. **Numeric placeholders** (`{rooms}`, `{adr}`) must be substituted before length-checking.

## Transformation Proposals — Budget Realism

Slide 5 (`transformationDescription`) and any "post-acquisition vision" copy MUST propose a transformation the property can actually afford. The budget envelope is:

```
budget_envelope = improvementsBudget + preOpExpenses        (from properties.*)
budget_floor    = 0.85 × budget_envelope                    (15% reserve for soft costs / overruns)
```

Anything you describe as part of the transformation must sum to ≤ `budget_floor` using the per-line cost ranges below. These are the same numbers in `hplus-renovation-benchmarks` — keep them in sync; if they drift, that is a bug per `agent-memory-files`.

| Upper-upscale + historic per key | per key | $140,000 – $250,000 | Full reno + MEP + preservation |

### Realism check (mandatory before finalising the proposal)

```ts
function isProposalRealistic(
  lines: Array<{ unit: number; lowUSD: number; highUSD: number }>,
  budgetFloor: number,
): { ok: boolean; lowSum: number; highSum: number } {
  const lowSum  = lines.reduce((s, l) => s + l.unit * l.lowUSD, 0);
  const highSum = lines.reduce((s, l) => s + l.unit * l.highUSD, 0);
  // Must fit at the LOW end of every range — otherwise the proposal needs the
  // most generous estimate of every line, which is not credible.
  return { ok: lowSum <= budgetFloor, lowSum, highSum };
}
```

**If `ok === false`:** drop the most expensive line, regenerate the proposal text, re-check. Repeat until it fits or only "soft-goods refresh" remains. Never publish a proposal that fails the check — the deterministic template fallback is preferred over an unrealistic vision.

### Allowed proposal vocabulary

When describing the transformation, use only items from the cost table above (or a clear synonym). Forbidden:

- "World-class spa", "destination restaurant", "five-star amenities" — vague, unbudgeted.
- Major land development (new buildings beyond ADUs, road work, new utilities to site) — outside the $-per-key model and routinely under-estimated.
- Branded partnerships ("Soho House style", "Auberge-level") — contractual, not a renovation line.

### LLM prompt addendum (append to `generatePropertyVisionText` prompt)

> Propose only transformations whose total low-end cost (using the cost table in the `hplus-vision-templates` skill) is ≤ ${budgetFloor}. Pick from: soft-goods refresh, en-suite key reno, add-a-key conversion, ADU, glamping, commercial kitchen, bar/café, gym, yoga studio, spa retrofit, pool, sauna/cold plunge, outdoor dining, hot tub / EV / smart-home bundle, historic preservation premium, MEP replacement. Do NOT mention items outside this list. Keep `transformationDescription` ≤ 200 characters.

def get_vision_templates(property_data: dict) -> dict:
    """
    Returns deterministic vision text for a property.
    Falls back to 'hotel' tier if type not recognized.
    """
    model = (property_data.get("businessModel") or
             property_data.get("hospitalityType") or "hotel").lower()
    tier = "retreat" if "retreat" in model else \
           "vrbo"    if model in ("vrbo", "vacation_rental", "airbnb") else \
           "hotel"
    tmpl = TEMPLATES[tier]
    return {k: v(property_data) for k, v in tmpl.items()}

## LLM Prompt Guidelines (for property-vision.ts)

When the LLM generates vision text, it MUST:
1. Use the template outputs as the **format reference** — match field lengths, capitalization style, and tone exactly
2. **Cite actual property data**: ADR, room count, purchase price, location — no generic placeholders
3. **Never use**: "exciting opportunity", "strong fundamentals", "world-class", "unique"
4. **Always use**: specific numbers, measurable outcomes, named demand drivers (e.g., "Belleayre ski area", "2.5hr NYC drive-market")
5. **Cinematic caption format**: `"[FEATURE] · [FEATURE]"` — ALL CAPS, concise, visual (max 2 features)
6. **Badge text**: 3–5 words, ALL CAPS, describes the experience brand promise

## Renovation Rationale Text by Scope

```python
RENOVATION_RATIONALES = {
    "light":   "Cosmetic refresh targeting quick-to-market timeline: design, photography, and OTA optimization.",
    "medium":  "Strategic renovation to unlock key revenue channels: en-suite upgrades, common area refresh, and F&B infrastructure.",
    "full":    "Full conversion buildout: en-suite guestrooms, commercial kitchen, spa amenities, and direct booking infrastructure.",
    "historic": "Historic preservation-grade renovation balancing character retention with modern hospitality standards.",
}

def get_renovation_rationale(scope: str, is_historic: bool) -> str:
    if is_historic:
        return RENOVATION_RATIONALES["historic"]
    return RENOVATION_RATIONALES.get(scope, RENOVATION_RATIONALES["medium"])
```
