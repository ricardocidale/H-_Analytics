## What CC discovered

Reading `server/ai/synthesis-schema.ts` line 47, `SynthesisOutputSchema` enforces a strict `z.enum(CANONICAL_RESEARCH_FIELDS)` constraint on the `field` key of every numeric value Opus emits during synthesis. That enum currently contains ~40 property-research keys: `adr`, `adrGrowth`, `occupancy`, `rampMonths`, `capRate`, `landValue`, `costFB`, `costMarketing`, `incentiveFee`, etc.

**None of the 5 funding keys appear in the enum:** `runwayBufferMonths`, `sizingOvershootPct`, `trancheGapMonths`, `revenueRampDelayMonths`, `burnFlexDownPct`.

This means **Option A (adapter shim wrapping `orchestrateResearch()`)** as originally drafted is structurally infeasible without modifying shared cognitive infrastructure:

- `synthesis-schema.ts CANONICAL_RESEARCH_FIELDS` would need to grow by 5 keys
- The synthesis system prompt in `server/ai/research-orchestrator.ts` (lines ~270–328) would need to teach Opus about funding-domain reasoning (LP capital appetite, runway buffer norms, tranche gap economics, ramp delay realism, burn flex headroom) — NEW content the prompt has never carried
- Every existing property-research run would now operate against a wider enum, with no behavioral change to property-research itself, but the prompt enlargement is a real cross-cutting modification

That's not a 200-line adapter; it's a meaningful expansion of the shared synthesis surface.

## The architectural fork

Two clean paths forward, plus a third hybrid:

### Option B (per-Specialist purpose-built pipeline)

Build a **separate** N+1 cognitive pipeline tailored to mgmt-co.funding:

- 3 funding-specific system prompts:
  - **Quantitative Panel (Gemini Flash):** "Given these capital-raise inputs (4 funding fields + 2 capital-raise dates) + LP comparables (canned for G1 v1, live API later) + the company's persona profile, what numeric ranges per dimension are reasonable? Output `{ runwayBufferMonths: { low, mid, high, reasoning }, ... } × 5 dimensions`"
  - **Market Panel (Claude Sonnet):** "Given current LP appetite + dry powder vintage + brand vertical positioning, what risk lens applies? Output qualitative reasoning per dimension."
  - **Synthesis Panel (Claude Opus):** "Reconcile A + B, output structured ranges with conviction per dimension. Schema = `FundingSynthesisOutputSchema` (NEW)."

- New `FundingSynthesisOutputSchema` purpose-built for the 5 funding keys
- New `MgmtCoFundingOrchestrator` class composing the three model calls
- Direct mapping to `RawVerdictDimension[]` via `DIMENSION_META`

**Pros:** Architecturally clean. Funding's structured output schema is exactly what funding needs; no enum gymnastics. Future per-Specialist pipelines (Revenue, Compensation, etc.) follow the same pattern.

**Cons:** ~300+ lines of new code. Three new system prompts (high-stakes design). Duplicates some panel-orchestration plumbing already in `research-orchestrator.ts` (parallel calls, model-vendor tracking, error boundaries, retry/regress). Pattern-divergence between property-research (uses shared orchestrator) and mgmt-co Specialists (each has its own) — a deliberate cost of separation.

### Option A-extended (extend shared synthesis to cover funding)

Modify shared infrastructure:

1. Add 5 funding keys to `CANONICAL_RESEARCH_FIELDS` in `synthesis-schema.ts`
2. Extend the synthesis system prompt in `research-orchestrator.ts` with funding-domain reasoning blocks
3. Extend the prompt builders (`research-prompt-builders.ts`) to produce funding-aware prompts when `type: "company"` AND specialistId is "mgmt-co.funding"
4. Build a thin adapter that reverses-maps the shared schema back to `RawVerdictDimension[]`

**Pros:** Reuses the existing N+1 pipeline. ~100 lines new code (the adapter). Future Specialists keep using the shared pipeline.

**Cons:** The shared synthesis prompt grows toward "knows everything about every domain" — death by accretion. Property-research runs now see funding keys in the schema (harmless but noise). The synthesis system prompt's enum-of-canonical-fields gets longer by 5 every Specialist that graduates. Eventually the prompt is too noisy for any one domain. Hard to escape without going to Option B later anyway.

### Option C (hybrid: pluggable synthesis schemas)

Refactor `orchestrateResearch()` to accept a `SpecialistSchema` parameter that determines which output schema + system prompt blocks are active for the run. Per-Specialist files contribute their own `<specialist>SynthesisSchema.ts` + prompt-block partial. The shared pipeline stays one entry point but specializes its prompt + output per Specialist.

**Pros:** Architecturally cleanest. Pipeline reuse + per-Specialist specialization. Future Specialists graduate by writing their schema + prompt-block, not their own pipeline.

**Cons:** Significant refactor of `research-orchestrator.ts` (591 lines) before any G1 work lands. Doctrine-level change requiring an ADR. Probably ADR-009 territory.
