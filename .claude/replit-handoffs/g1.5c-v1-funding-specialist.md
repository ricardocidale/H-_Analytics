## What v1 ships

A working Funding Specialist that, when the AnalystButton is pressed on the Funding tab:
1. **Server checks prerequisites** (hard-required fields against the saved `globalAssumptions` row). Missing → 400 with structured `missingFields[]`.
2. **Server invokes single-shot Opus call** with:
   - Funding-specific system prompt (Analyst persona; senior-advisor voice; range-first; investor-aware)
   - Rich context (full `CapitalRaiseInputs` + persona descriptor + benchmark ranges + 3 LP comparables + portfolio aggregate)
   - Strict Zod output schema (per-dimension `{low, mid, high, conviction, reasoning, evidenceRefs}`)
3. **Maps Opus output to `RawVerdictDimension[]`** using existing `DIMENSION_META`.
4. **Returns `AnalystVerdict`** via the existing `buildAnalystVerdict()` builder. ADR-008 meta fields populated: `cognitiveRunId` (from Opus run id), `vendorsUsed: ["anthropic"]`, `cacheState: "miss"` (cache wired in G6-P3).
5. **Client renders** the verdict on the Funding tab (range badges, conviction markers, reasoning text, evidence references).

Each becomes a follow-up packet that swaps in better behavior **without changing the function's contract** (`runFundingSpecialist(inputs, benchmarks, persona) → AnalystVerdict`). The contract stays stable; the body gets richer.

## Tasks

### S1 — Build funding-specific output schema

- **Files:** NEW `server/ai/specialists/mgmt-co-funding-output-schema.ts`
- **Change:** Zod schema `FundingSpecialistOutputSchema` enforcing:
  ```ts
  z.object({
    dimensions: z.array(z.object({
      key: z.enum(FUNDING_DIMENSION_KEYS),  // strict 5-key enum
      low: z.number(),
      mid: z.number(),
      high: z.number(),
      conviction: z.enum(["high", "moderate", "developing"]),
      reasoning: z.string().min(20).max(500),  // tight, specific
      evidenceRefs: z.array(z.number().int().min(0)).min(1).max(5),  // indexes into comparables
    })).length(5),  // exactly 5 dimensions
    overallNarrative: z.string().min(50).max(800).optional(),  // optional roll-up
  });
  ```
- **Acceptance criteria:**
  - [ ] TS clean
  - [ ] Schema parses representative test fixtures (3 personas: large-managementco, startup-boutique, expansion-stage)
  - [ ] Invalid outputs (4 dimensions, ranges where low>high, conviction outside enum) reject with clear errors
- **Test impact:** New unit test in S4 covers schema parsing edge cases.
- **Rollback:** Delete file.

### S2 — Build funding prompt module

- **Files:** NEW `server/ai/specialists/mgmt-co-funding-prompt.ts`
- **Change:** Two exported functions:

  export function buildFundingSystemPrompt(): string;
  export function buildFundingUserPrompt(

    benchmarks: AnalystWatchdogBenchmarks,
    persona: FundingPersonaContext,
    portfolioAggregate: PortfolioAggregate,
    comparables: readonly ComparableRow[],

- **System prompt content (CRITICAL — invest the time here):**
  - Analyst persona ("You are The Analyst, a senior advisor at Norfolk AI specializing in capital raises for boutique-luxury hospitality management companies. You have deep expertise in LP appetite, fund-of-funds dynamics, and brand-specific raise patterns.")
  - Voice rules (range-first; "your $30M raise" not "raises generally"; investor-aware "LPs in this vertical will ask about..."; no banned phrases per `branding-vocabulary-enforcement.md`)
  - Disagree when warranted ("If their inputs are weak, say so specifically. Don't hedge.")
  - Conviction calibration ("HIGH only when LP comp data + benchmark agreement; MODERATE when one signal supports; DEVELOPING when sparse")
  - Reasoning discipline ("Reference the user's specific numbers. Cite at least one comparable per dimension. Max 500 chars per dimension.")
  - Output format strict (the Zod schema dictated above)
- **User prompt content:**
  - Structured data dump: full inputs, persona descriptor (string), benchmark ranges with source labels, 3 LP comparables (full row data), portfolio summary (count, vintage, total raise need)
  - Final ask: "Output the verdict per the schema."
- **Acceptance criteria:**
  - [ ] TS clean
  - [ ] Prompt strings include all required context fields verifiably
  - [ ] No banned vocabulary per `branding-vocabulary-enforcement.md`
- **Test impact:** Snapshot test in S4 verifies prompt structure stable across edits.
- **Rollback:** Delete file.

### S3 — Build the runner function

- **Files:** NEW `server/ai/specialists/mgmt-co-funding-runner.ts`
- **Change:** Single async function:
  ```ts
  export async function runFundingSpecialist(
    inputs: CapitalRaiseInputs,
    benchmarks: AnalystWatchdogBenchmarks,
    persona: FundingPersonaContext,
    portfolioAggregate: PortfolioAggregate,
    deps: { anthropicClient: Anthropic },
  ): Promise<AnalystVerdict>;
  ```
  Body:
  1. Build system prompt + user prompt via S2 functions
  2. Call Opus (claude-opus-4-7) with `streamObject` from Vercel AI SDK + `FundingSpecialistOutputSchema`
  3. On parse success → map output dimensions to `RawVerdictDimension[]` using existing `DIMENSION_META` (per-key field mapping, evidence assembly from comparables[evidenceRefs])
  4. Build `AnalystVerdict` via `buildAnalystVerdict()`:
     - `tier: 1`
     - `meta: { cognitiveRunId: <opus run id>, vendorsUsed: ["anthropic"], cacheState: "miss" }`
     - Optional `voice.headline / detail` rendered by Voice Renderer downstream (NOT here — voice is a separate layer)
  5. On parse failure / Opus error → throw a typed error the route handler catches and degrades to Tier-0 fallback with `meta.fallbackReason: "tier1_temporarily_unavailable"`
- **Acceptance criteria:**
  - [ ] TS clean
  - [ ] Pure function — no DB writes, no global state mutation, no I/O beyond the LLM call
  - [ ] On stubbed Opus success → returns valid `AnalystVerdict` (passes `buildAnalystVerdict()` invariants)
  - [ ] On stubbed Opus throwing → throws typed `Tier1UnavailableError` (consumer-handled)
- **Test impact:** Comprehensive unit test in S4.
- **Rollback:** Delete file.
