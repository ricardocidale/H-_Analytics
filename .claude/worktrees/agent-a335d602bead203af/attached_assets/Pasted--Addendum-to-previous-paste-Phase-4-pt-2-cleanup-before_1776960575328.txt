● Addendum to previous paste — Phase 4 pt.2 cleanup before Phase 5a starts                        
                                                                                                  
  Three small items I flagged on Phase 4 that didn't make it into the                             
  Phase 5 paste. Phase 5 is now big enough (split 5a-e) that folding these                        
  into 5a would blow its scope. Clean them up as a standalone "Phase 4                            
  pt.2" commit FIRST, then start Phase 5a with a tight scope.                                     
                                                                                                  
  Estimated size: one commit, ~30 LOC, <30 min. Five gates between this                           
  and Phase 5a, so the two commits are independently reviewable.                                  
                                                                                                  
  1. sendBeacon auth cookies.                                                                     
     client/src/pages/admin/specialist/SpecialistPage.tsx:796
     navigator.sendBeacon fires the passive-ignore POST, but its cookie                           
     semantics differ from fetch. If an admin's session is expiring at                            
     navigation, the beacon may 401 silently and the calibration signal                           
     is lost. Add a test in tests/client/ that asserts sendBeacon POSTs                           
     ride the session cookie; or, if that's hard to test end-to-end, a                            
     runtime assertion in the beacon-send path that logs a warn if the                            
     response is non-204 (sendBeacon returns a boolean for queued-or-not,                         
     but the server-side log will show auth failures).                                            
                                                                                                  
  2. Dedup passive-ignore events.                                                                 
     React Strict Mode double-mounts in dev, and tab switches can fire                            
     multiple passive-ignore events for the same visible field. Table has                         
     no unique constraint on (specialist_id, field_key, actor_user_id,                            
     action). Fine for ratios at low volume; skews if noisy.                                      
                                                                                                  
     Easiest fix: client-side dedup via a ref holding a Set of                                    
     "${fieldKey}:${action}" keys that the unmount effect filters                                 
     against. Throw away the Set on specialistId change.                                          
                                                                                                  
  3. Rate limit recordRecommendationEvent route.                                                  
     server/routes/admin/specialists.ts:421                                                       
     POST /api/admin/specialists/:id/recommendation-event is auth-gated                           
     to admin but has no rate limit. If the client ever loops on error,
     it floods the table. This is a sibling of the 17 open CodeQL                                 
     js/missing-rate-limiting findings — closing it preemptively is                               
     $0 and shrinks the CodeQL backlog by one.                                                    
                                                                                                  
     express-rate-limit with a generous admin-tier window (e.g. 100 req/min                       
     per user) is plenty. Same middleware you'll use on the consolidated                          
     photos route in Phase 5a — install it once in 4pt.2, reuse in 5a.                            
                                                                                                  
  Order: 4pt.2 (this) → five gates → 5a (pipeline consolidation with the                          
  rate-limit middleware now already installed).
