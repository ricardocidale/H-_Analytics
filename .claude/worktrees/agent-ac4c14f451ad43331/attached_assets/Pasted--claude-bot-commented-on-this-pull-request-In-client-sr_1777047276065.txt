Extended reasoning...
What the bug is
client/src/components/admin/MarketRatesTab.tsx:74 declares const refreshRate = useRefreshRate() — a single shared useMutation instance whose isPending is a global boolean across the whole component. Inside the per-row render loop (lines 243–263) every row's button reads that same boolean twice:

Line 249: disabled={refreshRate.isPending} — every row's button disables when any row's mutation is in-flight.
Line 254: {refreshRate.isPending ? "Studying…" : "Analyst"} — every row's button swaps its label to "Studying…" when any row's mutation is in-flight.
client/src/lib/api/market-rates.ts:44-51 confirms useRefreshRate is a plain useMutation — it does not key pending state per rateKey.

The code path that triggers it
Admin clicks Analyst on row A → handleRefreshOne(rate.rateKey) runs refreshRate.mutate(rateKey).
React Query flips refreshRate.isPending to true globally.
Every other row re-renders. Each one reads the same refreshRate.isPending === true, so every button disables and every label becomes "Studying…".
The in-flight fetch is still only for row A's rate — nothing about rows B, C, D is actually being researched.
When row A's mutation settles, isPending flips to false everywhere; every row simultaneously re-enables and re-labels as "Analyst".
Why existing code doesn't prevent it
The useRefreshRate hook returns one mutation instance and shares it across renders. React Query doesn't — and can't — partition pending state by mutation variables unless the caller explicitly tracks it. The component has no such tracking. The sibling useRefreshAllRates uses the same shared-state pattern, but that's correct there: "Refresh all" IS a single global operation.

What the impact is
Two distinct harms:

User-visible mis-labeling (PR-introduced). Pre-PR the button was icon-only with a tooltip reading "Refresh this rate" — the shared-state still disabled every row, but the absence of a text label made the misbehavior invisible. This PR replaced the icon button with a labeled "Analyst"/"Studying…" button (diff @@ -243,15 +243,21), which makes the shared-state lie visible on every other row. The admin sees "Studying…" on rates they didn't click and reasonably concludes those rates are being fetched too — they aren't.

Parallel-refresh blocker (pre-existing). The admin cannot kick off a second rate's refresh while the first is in flight. Serialization is defensible UX for LLM-heavy operations, but the current implementation doesn't expose that intent — it just looks broken.

No data loss, no broken flow, no incorrect value gets written. Clicking any row after the in-flight completes works correctly.

How to fix
Track a per-key pending rateKey in local state and gate disabled + label on per-row equality:

const [pendingKey, setPendingKey] = useState<string | null>(null);

const handleRefreshOne = (rateKey: string) => {
  setPendingKey(rateKey);
  refreshRate.mutate(rateKey, {
    onSettled: () => setPendingKey((k) => (k === rateKey ? null : k)),
  });
};

// inside the row:
const isThisRowPending = pendingKey === rate.rateKey;
<Button
  disabled={isThisRowPending || (pendingKey !== null && pendingKey !== rate.rateKey)}
  data-testid={`button-analyst-${rate.rateKey}`}
>
  <IconSparkles className="w-3.5 h-3.5" />
  <span className="text-xs">{isThisRowPending ? "Studying…" : "Analyst"}</span>
</Button>;
If truly parallel refreshes are desired, replace pendingKey: string | null with pendingKeys: Set<string> and drop the pendingKey !== null && pendingKey !== rate.rateKey clause. If serialization is intentional, the above is sufficient — the label on non-pending rows stays "Analyst" and the tooltip still communicates the action.
