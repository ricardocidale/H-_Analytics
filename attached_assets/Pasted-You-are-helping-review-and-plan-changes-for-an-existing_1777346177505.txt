## Proposed target architecture to evaluate against

### A. Roles

There are only 3 intended roles:

* `super_admin`
* `admin`
* `user`

Legacy concepts such as `checker`, `investor`, `partner`, or similar should be treated as likely legacy concepts and should be flagged if still present in:

* code
* schema
* permissions
* route guards
* seeds
* UI copy
* docs
* business logic

But do not remove them automatically. First confirm how they are currently used.

### B. Base scenario and admin scenarios

There is a **seeded base scenario** created during development.

The proposed intent is that this base scenario is:

* the protected canonical property universe
* seeded/system-generated
* long-lived
* only destructively alterable by `super_admin`
* rarely deleted in whole or in part

`admin` and `super_admin` are expected to have effectively the same normal operating rights over scenarios, except that only `super_admin` can destructively alter the seeded base scenario.

At the admin level, scenarios are intended to be **visibility / assignment scenarios**, not financial modeling branches in the old sense.

An admin scenario is intended to:

* define which properties are visible in that scenario
* be derived from the base scenario
* never delete properties from the database
* act as a curated visibility set for assignment to users

All users are intended to be assigned either:

* a specific admin scenario, or
* the base scenario by default

If no explicit assignment exists, fallback should be the base scenario.

New properties created or uploaded by any user or admin are intended to:

* be added to the permanent canonical database
* automatically become part of the base scenario
* not automatically appear in all derived admin scenarios unless explicitly included there later

### C. User portfolios

At user level, the assigned admin scenario is intended to become the user’s **starting/base portfolio view**.

Users may create and save **portfolios** under different names.

This is intended to be a separate concept from admin scenarios.

A user portfolio may differ from the assigned scenario by:

* assumptions only
* visible properties only
* both

If the user edits anything from the assigned starting state, the user is intended to save those edits as a **new portfolio** or lose them.

The assigned starting scenario should therefore behave like a **read-only starting point**, not a mutable shared object.

At user level, removing a property is intended to mean hiding/removing it from that saved portfolio only, not deleting it from the database.

### D. Sharing portfolios

Users are intended to be able to share portfolios they own with other users.

The proposed sharing behavior is:

* owner enters the sharee’s email address manually
* the sharee email must already exist in the user database
* the app should not reveal user directory information to other users
* the sharer is expected to know the email from outside the app
* the app sends an email after the sharing request is made
* the app should not expose user search or identity lookup during sharing

The proposed clean sharing model is:

* original portfolio keeps a single owner
* shared users can view it
* shared users can create their own variation / fork / copy as their own portfolio
* shared users should not directly edit the original owner’s portfolio by default

`admin` and `super_admin` are intended to have full visibility into portfolios and sharing and may override if needed.

### E. User registration and profile rules

Users are intended to be created by admins.

Admin-created user fields are expected to include at least:

* name
* email
* login password
* role
* scenario assignment

Users are intended to be able to edit only these profile fields in **My Profile** on the front of the app:

* company name
* phone number
* job title

Users are not intended to edit their own:

* name
* email
* role
* assigned scenario
* password, unless there is a separate explicit password-change flow

Admins are intended to be able to edit all user information in **Admin > Users**.

All of this should be verified as server-side enforced, not only hidden in the UI.

### F. Property data governance

Properties are intended to be permanent canonical records.

Proposed rules:

* any user or admin can create/upload a property
* all new properties become part of the canonical database
* all new properties become part of the base scenario
* only `super_admin` can delete properties from the database
* user-level or portfolio-level removal is only visibility removal, not deletion

Prefer soft-delete / archive semantics over destructive deletion where feasible.

### G. Server-side priority

As much as possible should be implemented and enforced server-side.

The audit should specifically check whether the system currently relies too much on client-side logic for:

* permission enforcement
* scenario filtering
* portfolio ownership rules
* share permissions
* user-profile field restrictions
* property deletion protection
* scenario assignment resolution
* portfolio fork / duplication rules

Flag any sensitive logic that currently lives only in the client.

## Nomenclature and language audit

Audit terminology across:

* code
* schema
* UI
* seeds
* docs

Preferred naming direction:

* **Scenario** = admin visibility/assignment scenario
* **Portfolio** = user-owned saved working version
* **Base Scenario** = seeded canonical full property universe
* **Assigned Scenario** = admin-selected visibility scenario for a user
* **Base Portfolio** or **Assigned Base Portfolio** = user’s initial read-only starting point
* **Shared Portfolio** = portfolio visible to a non-owner through explicit sharing
* **Create Variation** / **Save as New Portfolio** = preferred wording for user-created derivatives

Flag likely-confusing or legacy terms such as:

* checker
* investor
* partner
* scenario when it really means portfolio
* delete when it really means hide/remove from view
* duplicate when it really means fork or clone-from-template

But recommend renaming only after understanding current usage.

* admins create users

Inspect whether sharing and notification behavior aligns with the proposed model.
