The code path that triggers it

async function runSchemaMigrations() {
  const { bootstrapDrizzleMigrationState, ... } = await import("./migrations/consolidated-schema");
  await bootstrapDrizzleMigrationState();
  const { migrate } = await import("drizzle-orm/node-postgres/migrator");

drizzle-orm's migrate() is journal-driven: readMigrationFiles opens _journal.json, iterates entries[], and applies each tagged SQL file. SQL files that exist on disk but are NOT listed in entries[] are silently skipped. That is stock drizzle behavior, not a local bug.

The argument "the PR also updated CI, so CI would catch this" does not save us. .github/workflows/ci.yml (changed in this PR, line ~95) runs:

There is no drizzle-kit check step, no duplicate-idx guard, and no count(00NN_*.sql) == entries.length assertion anywhere in the CI workflow. Nothing in this PR would have failed the 0020_ collision or the journal omission.
