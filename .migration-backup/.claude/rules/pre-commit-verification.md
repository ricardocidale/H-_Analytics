# Pre-Commit Verification — Mandatory

This rule exists because we kept finding 40+ bugs per audit sweep. Almost
every one would have been caught by running the existing verification
commands before committing. From now on, every commit either passes every
gate below or it does not land.

**This rule binds both agents and any human committer.**

## The five gates

No commit may be pushed to `main` (or any branch that will merge to `main`)
unless ALL of these pass:

| # | Command | Pass criterion |
|---|---|---|
| 1 | `npx tsc --noEmit` | Exit code 0. Zero type errors. |
| 2 | `npm run lint` | Exit code 0. Zero errors; zero warnings. |
| 3 | `npm run test:file -- tests/audit/vocabulary-compliance.test.ts` | 11/11 pass. |
| 4 | `npm run test:summary` | All test files pass. No skipped financial/proof tests. |
| 5 | `npm run verify:summary` | Opinion: **UNQUALIFIED**. |

All five must be run. All five must pass. There is no version of "I'll fix
that in a follow-up" that is acceptable — follow-ups are how bugs compound.

## When gates fail

**Stop.** Do not commit. Investigate root cause.

- If it's a genuine bug you just introduced → fix it and re-run.
- If it's a pre-existing failure you didn't cause → **escalate to Claude Code via a comment in the active handoff** (or a `BLOCKED.md` sibling for urgent cases). Do not skip the gate. Do not `--no-verify`. Do not delete the failing test.
- If the failure is cosmetic (e.g., a lint warning on an autogenerated file) → still pause; the rule is "zero warnings," not "zero warnings I think matter."

## Where to run

- **Replit Agent:** run in the sandbox terminal before every `git commit`.
- **Claude Code:** run via the Bash tool before every commit, including
  docs commits that touch anything importable by TypeScript (`.ts`/`.tsx`
  in the include set of `tsconfig.json`).
- **Multi-commit handoffs:** every commit in a multi-commit handoff must
  pass the full gate independently. A commit that "builds on a previous
  commit that will pass later" is a violation.

## Why this rule exists (historical record, not motivation)

Documented failures that would have been prevented by these gates:

- **Phase 2 vocab regression** (`ae563c1c`, April 18): `"Ask the Analyst"`
  literal shipped in a tooltip because the vocabulary test wasn't run
  pre-commit. Caught only on Replit's post-commit verification sweep.
  Cost: one BLOCKED handoff + a bug-fix commit.
- **Task #15 contract bug** (surfaced April 18): `PortfolioPropertySummary`
  was missing `isActive` while `PropertyFeeSummaryTable` rendered an
  "Excluded" badge off it. The `any[]` prop hid the missing field. A
  fully typed component would have failed TS compile and the gate would
  have caught it.
- **KB orphan directory** (`server/ai/kb/`, added `640e889f`): 19 files,
  ~900 lines, never wired into the RAG pipeline. No test flagged them.
  Lived in the repo unused for months until Phase 5B discovery.

These are not hypothetical. Each one cost real session time, real architect
reviews, real rework. This rule is the cheapest possible prevention.

## Relationship to other rules

- **`testing-strategy.md`** defines WHICH tests exist and when they're required.
  This rule says: **run them before every commit, not "when you remember."**
- **`claude-replit-split.md` §Guardrails #4** already requires type-check + tests
  after every commit. This rule upgrades that from guardrail to blocking gate,
  and specifies the exact commands.
- **`documentation.md` §After Bug Fixes** lists `npm run test:summary` +
  `npm run verify:summary`. This rule makes those two commands part of the
  baseline for EVERY commit, not just bug fixes.

## Enforcement

`tests/proof/rule-compliance.test.ts` should flag commits that bypass this
rule (e.g., by checking for `--no-verify` in git reflog or missing
verification references in commit messages). Adding that enforcement is
itself an upcoming task.

For now, enforcement is honour + commit-message discipline:

- Commit messages for non-docs commits SHOULD include a line like
  `Verified: TS 0, Lint 0, Vocab 11/11, test:summary PASS, Verify UNQUALIFIED`
  so reviewers can confirm the gate ran.
- A commit without this line is not necessarily in violation, but it is
  less trustworthy and should draw extra review.
